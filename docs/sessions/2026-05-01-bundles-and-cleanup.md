## Decisions captured

### Step 3: placeholder-UUID safety + impure-mode requirement

The scenario this guards against: a host has a placeholder
`hardware-configuration.nix` (the all-zeros sentinel) and someone runs
`sudo nixos-rebuild switch --flake .#<host>` on the wrong machine.
Without the assertion, the build succeeds and produces an unbootable
system; with it, eval aborts unless `NIXOS_ALLOW_PLACEHOLDER=1` is
present.

The assertion lives on `system.build.toplevel`, so:

- `nix build .#nixosConfigurations.<host>.config.system.build.toplevel` aborts
- `nixos-rebuild switch` aborts
- `nix flake check` (pure) walks every entry in `nixosConfigurations`
  built-in CLI behavior we can't suppress, so plain pure check still
  evaluates the assertion.

The fix is two-part:

1. The auto-check at `flake-modules/nixos.nix:62-69` filters out
   placeholders so the per-system `checks.<system>.configurations:nixos:<host>`
   entries don't include them — that's why `nix flake check` doesn't
   crash on the *build* of placeholders.
2. But pure `nix flake check` still walks the `nixosConfigurations`
   attrset and forces the assertion. So the documented operator command
   for any session involving placeholder hosts is now:

```sh
NIXOS_ALLOW_PLACEHOLDER=1 nix flake check --impure
```

AGENTS.md was updated with this in step 3.

### Step 4: `flake.lib.mkPkgs` works as a single definition

`mkPkgs system` builds an arch-pinned nixpkgs instance with overlays +
allowUnfree + allowAliases=false. Three host bridges had identical copies
inline (pb-x1, family-laptop, wsl, ah-1). Extracted to one place.

Mechanically: only one module writes `flake.lib`, so no explicit
`options.flake.lib` declaration is required at this step. Step 5 made
that no longer true (bundles add `flake.lib.bundles.homeManager.*`),
which forced step 5 to add `flake-modules/lib.nix` declaring the option
as a freeform-typed submodule.

### Step 5: bundles + an unintended polkit fix

Four bundles published as `flake.lib.bundles.homeManager.<name>`:

- `base` (7): btop, direnv, gh, git, neovim, tmux, zsh
- `dev` (9): base + ai-cli + build-deps
- `desktop` (22): dev + 13 niri/quickshell/audio/etc.
- `kid` (13): parallel low-trust set (chromium-managed, zoom; no chrome/dev tools)

Bundles are plain lists of module paths. Hosts consume them by
list-concatenation, so per-host extras just append:

```nix
imports = config.flake.lib.bundles.homeManager.desktop ++ [
  config.flake.modules.homeManager.audio   # pb-x1-only audio presets
];
```

**Why `flake-modules/lib.nix` was needed:** flake-parts marks every
`flake.<X>` (and every leaf below it) as `types.unique` by default,
meaning only one module can define each path. Once both `mk-pkgs.nix`
*and* every bundle file write under `flake.lib`, an explicit option
declaration with mergeable types becomes mandatory. The lib.nix file
declares `options.flake.lib` as a `submodule` with `freeformType =
lazyAttrsOf raw` and a typed `bundles.homeManager` sub-attrset of
`lazyAttrsOf (listOf raw)`.

**The polkit fix (incidental but real):** `polkit-agent.nix` and
`quickshell.nix` both wrote into `programs.niri.settings.spawn-at-startup`
at `lib.mkAfter` (priority 1500), leaving import order to break ties.
The bundle refactor surfaced this because the kid bundle ended up
spawning quickshell before polkit-agent. Fixed by changing
`polkit-agent.nix` to `lib.mkOrder 1490` so the agent fires first
deterministically regardless of caller import order.

**Byte-equivalence outcome:**

- All 4 NixOS toplevels: byte-identical to pre-refactor.
- `nas@ah-1` HM closure: byte-identical.
- 5 of 6 HM closures changed drv hash, all functionally equivalent:
  - `m@`, `s@family-laptop`: polkit ordering FIX (intended) + cosmetic
    `home.packages` JSON insertion-order reorder.
  - `p@pb-x1`, `p@family-laptop`, `p@wsl`: cosmetic `home.packages` JSON
    insertion-order only; `~/.nix-profile` contents unchanged at runtime
    (nix-env merges by priority, not list order).

Diffed niri `config.kdl` between polkit-only-fix-kid and full-bundle-kid:
identical. The only semantic delta in the entire refactor is the
polkit/quickshell ordering, and that's a fix.

### Step 8: `nh` + `nvd` + `nom`, with the NH_FLAKE gotcha

`nh` wraps `nixos-rebuild` / `home-manager switch` / `nix-collect-garbage`
and pipes through `nom` for structured progress and `nvd` for closure
diff. The whole point is so common workflows become `nh os switch` and
`nh home switch <user>@<host>` from anywhere inside the repo.

**The trap I fell into:** first attempt set
`NH_FLAKE = toString ../..;` as a derivation env attr in the `mkShell`.
Inside the dev-shell `echo $NH_FLAKE` printed `/nix/store` — because
`toString ../..` is evaluated at flake evaluation time relative to the
store copy of `flake-modules/dev-shell.nix`, which lives at
`/nix/store/<hash>/flake-modules/dev-shell.nix`, so `../..` is
`/nix/store/<hash>` → `/nix/store/<hash>/..` → `/nix/store`.

Fix: set it in `shellHook` at runtime:

```nix
shellHook = ''
  ${pre-commit-check.shellHook}
  export NH_FLAKE="$(git rev-parse --show-toplevel 2>/dev/null || pwd)"
'';
```

This resolves to the actual workspace root (`/home/p/nixos`) regardless
of which subdirectory the dev-shell was opened from.

### Step 9: sops scaffolding was 100% security theater

What was there:
- `flake-modules/secrets.nix` — option-typed dispatcher with
  `flake.modules.{nixos,homeManager}.secrets` modules importing
  `inputs.sops-nix.{nixosModules,homeManagerModules}.sops`. Every host
  guard was `lib.mkIf (cfg.X != null)`.
- `.sops.yaml` — recipient list with `age1REPLACEME_PERSONAL...`
  placeholders.
- `secrets/README.md` — bootstrap instructions.
- `flake.nix` — `inputs.sops-nix` declared.
- 4 host bridges — commented-out `# secrets = { ... };` blocks all
  prefixed "Uncomment after running the bootstrap...".
- `flake-modules/git.nix` — `identityFile` option that depended on
  `config.sops.secrets.<name>.path` paths.
- `flake-modules/ai-cli.nix` — multi-paragraph commented example of
  how to wire OPENAI_API_KEY through sops.

What was NOT there:
- A real age key on any machine.
- A real recipient in `.sops.yaml`.
- An encrypted `secrets/common.yaml`.
- Any host that imported `config.flake.modules.{nixos,homeManager}.secrets`.
- Any consumer of `git.identityFile`.

So the entire stack was inert at runtime: it contributed zero bytes to
any built closure and existed solely as documentation of an unimplemented
intent. AGENTS.md said "Do not add nix-darwin, a secrets framework
(sops-nix/agenix), or move configs to a separate dotfiles repo unless
asked" — meaning sops being there at all already violated the stated
rules. README.md still said "No secrets module yet" while the code said
"Yes there is, kinda".

User decision: rip it out cleanly. A real secrets framework can be
reintroduced when there's an actual secret to protect (homelab service
creds, Tailscale auth keys, SSH CA private key) and a clear trust
model. The previous design's ambivalence between SSH-host-key
derivation, per-user age keys, and per-machine age keys was itself a
sign that the design hadn't been pinned down.

The trust-model conversation that preceded the rip-out is worth
preserving for whenever this comes back. Three options were on the
table:

- **A. Per-machine SSH-host-key derivation** (mainstream sops-nix
  idiom): NixOS already generates `/etc/ssh/ssh_host_ed25519_key`,
  `ssh-to-age` converts it to an age private key, sops-nix reads it
  directly. Zero manual key management once bootstrapped. Doesn't work
  for WSL (no real sshd) or for HM-only flows.
- **B. Per-user manual age key**: `~/.config/sops/age/keys.txt`,
  generated once per `(user, machine)`, never committed. HM-side sops
  reads it. Trust = whoever has the file.
- **C. Hardware-backed (YubiKey)** via `age-plugin-yubikey`. Strong
  defense against repo + machine compromise; requires 2+ tokens for
  redundancy; provisioning takes 10–20 min per host.

The recommendation was **hybrid A+B**: SSH-host-key for hosts that have
real sshd (pb-x1, family-laptop, ah-1), per-user age keys for
HM-side secrets and for WSL where NixOS-side sops is awkward. But the
user chose to defer entirely until there's a real secret to protect.

Net change for step 9: 13 files touched, 49 insertions, 484 deletions.
All 10 closures byte-identical to step 8 — proof the scaffolding had
never affected anything that gets built.

## Process notes worth keeping

### Workflow per step

1. Make the change.
2. `nix fmt` (the formatter is `nixpkgs-fmt` exposed via `flake.formatter`).
3. `git add -A`.
4. `NIXOS_ALLOW_PLACEHOLDER=1 nix flake check --impure`.
5. Smoke-build all 10 closures in one parallel `nix build` invocation.
6. Compare store hashes against the previous commit's hashes (mental
   diff or eyeball — they're either identical or they're not).
7. If hashes changed unexpectedly, find out why before committing.
8. Commit with a message that explains *why* the change was made and
   *what regression criterion* was used (closure byte-equivalence in
   every step here).

### What "all 10 closures" means

Not 7 (= what `nix flake check` actually builds on x86_64-linux on the
dev box), but all 10 that the flake produces:

- 4 NixOS toplevels: pb-x1, family-laptop, wsl, ah-1
- 6 HM activation packages: p@pb-x1, p@family-laptop, m@family-laptop,
  s@family-laptop, p@wsl, nas@ah-1
- (wsl-arm is aarch64 and skipped on the dev box; would need
  cross-compile or a real ARM machine to validate.)

Use `--impure` to include placeholders (family-laptop, ah-1).
