# 2026-04-29 — Battery management: charge thresholds, hibernate, power-saver

## Goal

Three connected pieces of laptop battery management on `laptop` (Lenovo
X1 Yoga gen 7):

1. **Cap charging at 80%** (resume at 75%) to extend pack lifespan.
2. **Hibernate at 10% battery** (UPower `CriticalPowerAction = Hibernate`).
3. **Switch power-profiles-daemon to `power-saver` at 40%** while
   discharging, restoring the prior profile when AC is reconnected or
   the battery climbs back above the threshold.

User preferences locked in:

- **Hibernate swap**: full-RAM 32 GiB swapfile (rejected the 16 GiB
  half-RAM compromise; full RAM means we can hibernate even with all
  31 GiB used).
- **Power-saver trigger lives in the existing `idled` daemon**, not a
  new poller or standalone systemd unit. We already have a Rust user
  daemon with `zbus` listening to logind on this host; adding a UPower
  + PPD watcher there keeps the surface area small and reuses the
  existing graphical-session lifecycle.
- **Charge thresholds via systemd-tmpfiles**, not TLP. TLP and
  power-profiles-daemon cannot coexist; PPD is already enabled, so we
  drive sysfs directly. Lenovo X1 Yoga gen 7 exposes
  `charge_control_{start,end}_threshold` on a stock kernel.

## Context

Pre-existing on this host:

- `services.upower.enable = true` (turned on for the quickshell battery
  chip in the prior session).
- `services.power-profiles-daemon.enable = true` (defaulted by
  `modules/nixos/desktop/niri.nix`).
- No swap, no `resume=` kernel param. `/swap/` did not exist.
- The `idled` user daemon (introduced in `2026-04-29-idle-lock-fix.md`)
  already runs as a systemd-user service with `zbus`, `tokio`, and a
  logind connection.

The choice tree was:

| trigger / sink | option A | option B (chosen) |
|---|---|---|
| switch to power-saver at 40% | poll battery from a shell script in a systemd timer | extend `idled` with a UPower watcher + PPD setter |
| hibernate at 10% | write a custom systemd-logind handler | UPower's built-in `CriticalPowerAction` |
| charge cap at 80% | TLP (would conflict with PPD) | systemd-tmpfiles writes to sysfs |
| swap for hibernate | 16 GiB (covers most working sets) | 32 GiB (covers all of RAM) |

## Implementation

### `modules/nixos/battery.nix` (new)

Single-host-friendly module gated on `variables.battery.enable`. Three
concerns:

1. **Charge thresholds** via `systemd.tmpfiles.rules` writing
   `w+ /sys/class/power_supply/BAT0/charge_control_{start,end}_threshold`.
   `w+` truncates and writes; if the sysfs file is missing (non-Lenovo
   hardware) it no-ops silently. Order matters: write `start` first,
   then `end`, because on some kernels you can't lower `end` below the
   current `start`.

2. **Btrfs swapfile** via `swapDevices = [{ device = "/swap/swapfile";
   size = 32*1024; }]`. NixOS handles `chattr +C` on the parent dir and
   creates the file on activation. Per the kernel docs
   (https://btrfs.readthedocs.io/en/latest/Swapfile.html) a btrfs
   swapfile must be on a non-CoW subvolume, never snapshotted, and a
   single contiguous extent — NixOS's swap activation script gets
   the first two right, and the file's freshly created so it's
   contiguous by construction.

3. **Hibernate boot params**:
   - `boot.resumeDevice = "/dev/disk/by-uuid/e2ac9790-…"` — hard-coded
     UUID of the btrfs root. UUID over `/dev/nvme0n1p2` so nvme
     renumbering doesn't break resume.
   - `boot.kernelParams = [ "resume_offset=0" ]` — placeholder; see the
     UX wart below.
   - `systemd.services.battery-resume-offset` — oneshot after
     `swap.target` that runs `btrfs inspect-internal map-swapfile -r
     /swap/swapfile`, compares to `/proc/cmdline`, and prints a stderr
     nag if stale, telling the user exactly what to put in
     `boot.kernelParams` and reboot once.

4. **UPower critical action** via `environment.etc."UPower/UPower.conf"`:
   ```ini
   [UPower]
   UsePercentageForPolicy=true
   PercentageLow=20
   PercentageCritical=15        # critP + 5
   PercentageAction=10          # critP
   CriticalPowerAction=Hibernate
   ```
   `Critical` is set 5% above `Action` so the desktop has a chance to
   show the "battery critical" notification before UPower itself
   triggers hibernate.

#### Hibernate first-boot UX wart (intentional)

NixOS does not auto-derive `resume_offset` for swapfiles. The chain is:

1. First activation creates `/swap/swapfile`.
2. Initrd on the next boot has `resume_offset=0` from kernelParams →
   wrong → hibernate-resume would silently boot fresh.
3. `battery-resume-offset.service` runs, reads the real offset, prints
   a one-line nag.
4. User edits `hosts/laptop/variables.nix` (or the kernelParams
   directly), `nixos-rebuild switch`, reboot.
5. Hibernate-resume now works on subsequent boots.

We rejected automating step 4 (would require a second activation pass
or a writable nix module). The nag is loud enough that the wart costs
one reboot, ever.

### `packages/idled/src/power.rs` (new) + `main.rs` glue

New module under `idled` that:

- Connects to UPower via `zbus`, proxying
  `org.freedesktop.UPower.Device` on the `DisplayDevice` aggregate
  battery (works regardless of how many physical cells / UPS / mice
  with batteries the system reports).
- Connects to power-profiles-daemon via two proxies — the new
  `org.freedesktop.UPower.PowerProfiles` name (PPD ≥ ~0.20) and the
  legacy `net.hadess.PowerProfiles` — and picks whichever one
  responds. Wrapped in an internal `Ppd` enum so the policy code
  doesn't care which is live.
- Subscribes to `Percentage` and `State` PropertiesChanged signals.

State machine in `evaluate()`:

- **Descent**: `state == Discharging && pct <= threshold && snapshot ==
  None` → snapshot the current `ActiveProfile`, set `power-saver`. If
  the user is *already* on `power-saver` we record `"balanced"` as the
  restore target so reconnecting AC still recovers a useful profile.
- **Ascent**: snapshot present AND (`charging || state == FullyCharged
  || pct >= threshold + hysteresis`) → check that `ActiveProfile` is
  still `power-saver` (don't override the user manually picking
  `performance`); if so, restore the snapshot, then clear it.
- **Hysteresis** is a separate config knob (default 5%) so you don't
  flap between profiles when the reported percent oscillates around
  the threshold.

`Config` in `main.rs` gains an optional `[battery]` section:

```toml
[battery]
power_saver_percent = 40
hysteresis = 5
```

Absence of the section disables the watcher entirely (clean opt-in
for hosts without a battery or without PPD). Errors connecting to
UPower or PPD are logged and the watcher task exits — they do not
take down the rest of `idled`.

### Wiring

- `hosts/laptop/variables.nix` gains a `battery = { … }` block (all
  six knobs documented inline).
- `modules/nixos/default.nix` imports `./battery.nix`.
- `modules/home/desktop/idle.nix` reads `variables.battery.*` and
  appends a `[battery]` block to the generated TOML when both
  `battery.enable` is set and `powerSaverPercent > 0`.

### Why a Rust dbus watcher and not `powerprofilesctl` / shell

- `powerprofilesctl` exists but forking a shell on every UPower signal
  is wasteful when we already have a connected zbus session.
- Reading the previous `ActiveProfile` to restore it cleanly is dbus
  property-get, not a shell-friendly operation.
- Hysteresis state would have to live in a tmpfile or env-of-a-running-
  process either way; a long-lived task that just owns the
  `Option<String>` snapshot is straightforward.

## Open / future

- The first-boot resume_offset dance is documented but unautomated.
  If hibernate-resume becomes important enough on multiple hosts, a
  second-pass activation script could write the offset back to
  `boot.kernelParams` via an out-of-band mechanism (etc-overlay
  managed file read at next eval). Not worth the complexity today.
- The `[battery]` watcher only handles PPD switching. If we ever
  want CPU governor pinning, screen-dim curves, or radio shutoffs at
  low battery, those should also live in `idled` (one daemon per
  policy domain, not one daemon per knob).
- We don't try to detect non-laptop hardware. The whole module is
  gated on `variables.battery.enable`; desktop hosts simply don't
  set it.
